# pr-feedback-triage

PR レビュー指摘を「指摘されたから直す」のではなく**技術的根拠**に基づいて判定する。Copilot でも人間でも基準は同じ。判定結果と対応内容は PR の元コメントへ**スレッド返信**として残し、レビュアー側でも追跡可能にする。

## 大原則

- **権威ではなく根拠で判断する** — 「コードがどう壊れるか／どう改善されるか」を具体的に説明できなければ却下。
- **忖度禁止** — 受諾／却下とも理由を明示する。却下を恐れて不要な変更を入れない。
- **指摘されたスコープを超えない** — 関連する別件のリファクタを混ぜない。

## 分類タクソノミー

各指摘に 1 ラベル付ける。判定（受諾／却下）と組み合わせて表記。

| ラベル | 意味 |
|:------|:-----|
| `BUG` | 動作不具合・データ破損・例外漏れ |
| `SEC` | セキュリティリスク（認可漏れ、injection、シークレット露出） |
| `ROBUST` | 防御的検証・エラーハンドリング・契約整合 |
| `DOC` | コメント／ドキュメントとコードの不一致 |
| `TEST` | テスト欠落・不正確さ |
| `PERF` | 体感できるパフォーマンス問題 |
| `STYLE` | 嗜好・命名・微細リファクタ |
| `YAGNI` | 仮想的将来拡張への過剰設計要求 |
| `COMPAT` | 後方互換シムの追加要求 |

## 判定の対応関係

| 判定 | 対象 | 行動 |
|:----:|:-----|:-----|
| **A. 受諾** | `BUG` / `SEC` / `ROBUST` / `DOC` / `TEST` / `PERF` で根拠が具体的なもの | 修正コミット + 返信 |
| **B. 部分受諾** | 問題意識は正しいが提案が広すぎる／ズレている | 必要な部分だけ修正 + 受諾範囲と却下範囲を返信 |
| **C. 却下** | `STYLE` / `YAGNI` / `COMPAT`、または前提誤読・既存規約と整合する選択 | 修正せず、却下理由を返信 |
| **D. 保留** | 影響範囲が広い・既存設計の前提に踏み込む | ユーザーに判断を仰ぐ |

却下に該当しやすい例: 命名・early return vs if-else、削除済み機能の deprecation path、自明な内容のコメント追加、カバレッジ稼ぎだけのテスト。

## 実行手順

### 1. 指摘を取得

```bash
gh pr view <PR#> --json title,headRefName,baseRefName,url
gh pr view <PR#> --comments                                   # review summary
gh api repos/<owner>/<repo>/pulls/<PR#>/comments              # inline comments (本体)
```

各 inline コメントから `id` / `path` / `line` / `body` を抽出。`user.type == "Bot"` で自動レビュアーを識別。

### 2. 各指摘を分類 + 判定

```
[#1] internal/config/validate.go:218
  指摘: `[home_files].files` で `:` を拒否すべき
  分類: ROBUST
  判定: A (受諾)
  根拠: compose.go:169 が短縮形 `source:target` で結合する。`:` は volume 仕様を破壊する。
  対応: validate.go と assertSafeHomeRel の両方に追加 + テスト追加
```

### 3. 受諾分のみ修正

- A / B 判定の指摘ごとに、論理的にまとまった単位でコミット分割。
- `just ci` を**必ず**実行してグリーン確認。
- CHANGELOG 判定: 「エンドユーザーまたはプラグイン作者の操作・設定・動作が変わるか？」→ Yes なら `Unreleased` に記載、No なら何もしない。

### 4. PR コメントへ返信を投稿（必須）

判定結果は GitHub の元コメントに**スレッド返信**として残す。受諾／却下とも投稿。

返信書式（共通）:

```
判定: <分類> (受諾 | 却下 | 部分受諾)
根拠 / 理由: <なぜそう判定したか>
対応: <何をどう変更したか> ※受諾 / 部分受諾のみ
コミット: <short SHA>           ※受諾 / 部分受諾のみ
```

部分受諾の場合は `対応` を `受諾範囲:` / `却下範囲:` の 2 行に分ける。

投稿コマンド:

```bash
gh api -X POST \
  "repos/<owner>/<repo>/pulls/<PR#>/comments/<COMMENT_ID>/replies" \
  -f body="$(cat <<'EOF'
判定: ROBUST (受諾)
根拠: ...
対応: ...
コミット: 9102251
EOF
)"
```

### 5. ユーザー向けサマリ報告

PR 投稿とは別に、表形式でローカル報告する。

```
| # | ファイル:行 | 分類 | 判定 | 概要 | コミット |
```

末尾に CI ステータス・コミット一覧・返信投稿の有無を併記。
